## Project Overview

### Goal
Build a POC for a configuration-based workflow system using Camunda with dynamic form generation capabilities.

### Key Features
- **Dynamic Workflow Configuration**: Workflows defined in database, BPMN generated automatically
- **Workflow Versioning**: In-flight processes continue on their version, new processes use new version
- **Dynamic Forms**: Forms rendered from database configuration with 25+ field types
- **Conditional Logic**: Show/hide fields based on other field values
- **Parallel Workflows**: Support both sequential and parallel approval paths
- **Admin Override**: Manual workflow progression when needed
- **Audit Trail**: Complete history of workflow steps and field changes

### Architecture
- **Monorepo**: Single project with embedded React frontend
- **Backend**: Spring Boot + Camunda (embedded engine)
- **Frontend**: React + TypeScript served as static content from Spring Boot
- **Database**: H2 in-memory (for POC)
- **Build**: Gradle with React build integration

## Setup Instructions

### Prerequisites
- JDK 17 or higher
- Node.js 20.x or higher
- Gradle 8.x (or use wrapper)

### Initial Project Setup

1. Create Gradle project with Spring Initializr dependencies
2. Configure Gradle to build React during Spring Boot build
3. Set up H2 database configuration
4. Configure Camunda embedded engine
5. Set up OpenAPI/Swagger UI

### Gradle Configuration

**build.gradle** should include:
- Spring Boot plugin
- Node Gradle plugin (for React build)
- Jacoco plugin (code coverage)
- Custom tasks: npmInstall, npmBuild, copyReactBuild, integrationTest, testAll
- Dependencies: Spring Boot, Camunda, H2, OpenAPI, Lombok, MapStruct, testing libraries

**Key Gradle Tasks**:
```bash
./gradlew bootRun          # Run backend only (dev mode)
./gradlew build            # Build everything (backend + React)
./gradlew test             # Run unit tests
./gradlew integrationTest  # Run integration tests
./gradlew testAll          # Run all tests (unit + integration + frontend)
./gradlew jacocoTestReport # Generate coverage report
```

### Application Configuration

**application.yml**:

## Phase 1: Core Workflow Engine

### Goals
- Set up database schema for workflow configuration
- Implement BPMN generation from database configuration
- Deploy generated BPMN to Camunda engine
- Create REST APIs for workflow management
- Start and manage workflow instances

### Database Schema

Create **schema.sql** with the following tables:

#### WORKFLOW_TYPE
```sql
CREATE TABLE workflow_type (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(255) NOT NULL,
    description TEXT,
    current_version_id BIGINT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

#### WORKFLOW_VERSION
```sql
CREATE TABLE workflow_version (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    workflow_type_id BIGINT NOT NULL,
    version_number INT NOT NULL,
    process_definition_key VARCHAR(255),  -- e.g., 'purchase-request-v1'
    camunda_deployment_id VARCHAR(255),
    bpmn_xml TEXT,  -- Store generated BPMN for auditing
    status VARCHAR(50) NOT NULL,  -- DRAFT, ACTIVE, DEPRECATED
    effective_from TIMESTAMP,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    created_by VARCHAR(255),
    FOREIGN KEY (workflow_type_id) REFERENCES workflow_type(id),
    UNIQUE(workflow_type_id, version_number)
);
```

#### WORKFLOW_STEP
```sql
CREATE TABLE workflow_step (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    workflow_version_id BIGINT NOT NULL,
    step_key VARCHAR(100) NOT NULL,  -- e.g., 'SPONSOR_APPROVED'
    step_name VARCHAR(255) NOT NULL,
    step_order INT NOT NULL,
    step_type VARCHAR(50) NOT NULL,  -- USER_TASK, SERVICE_TASK, START_EVENT, END_EVENT
    assignee_type VARCHAR(50),  -- REQUESTOR, CONTACT_ROLE, USER, EXPRESSION
    assignee_value VARCHAR(255),  -- e.g., 'SPONSOR' for contact role
    config_json TEXT,  -- Additional configuration as JSON
    FOREIGN KEY (workflow_version_id) REFERENCES workflow_version(id),
    UNIQUE(workflow_version_id, step_key)
);
```

#### WORKFLOW_STEP_CONDITION
```sql
CREATE TABLE workflow_step_condition (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    from_step_id BIGINT NOT NULL,
    to_step_id BIGINT NOT NULL,
    condition_type VARCHAR(50),  -- SEQUENTIAL, PARALLEL_FORK, PARALLEL_JOIN
    condition_order INT,
    FOREIGN KEY (from_step_id) REFERENCES workflow_step(id),
    FOREIGN KEY (to_step_id) REFERENCES workflow_step(id)
);
```

#### WORKFLOW_INSTANCE
```sql
CREATE TABLE workflow_instance (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    workflow_type_id BIGINT NOT NULL,
    workflow_version_id BIGINT NOT NULL,
    camunda_process_instance_id VARCHAR(255),
    business_object_type VARCHAR(100),  -- e.g., 'PURCHASE_REQUEST'
    business_object_id VARCHAR(255),     -- e.g., 'PR-001'
    current_step_key VARCHAR(100),
    status VARCHAR(50) NOT NULL,  -- IN_PROGRESS, COMPLETED, CANCELLED, ERROR
    started_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    completed_at TIMESTAMP,
    FOREIGN KEY (workflow_type_id) REFERENCES workflow_type(id),
    FOREIGN KEY (workflow_version_id) REFERENCES workflow_version(id)
);
```

#### WORKFLOW_INSTANCE_HISTORY
```sql
CREATE TABLE workflow_instance_history (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    workflow_instance_id BIGINT NOT NULL,
    step_key VARCHAR(100) NOT NULL,
    action VARCHAR(50) NOT NULL,  -- STARTED, COMPLETED, APPROVED, REJECTED, REWORK, MANUAL_OVERRIDE
    actor_user_id VARCHAR(255),
    actor_user_name VARCHAR(255),
    comments TEXT,
    is_manual_override BOOLEAN DEFAULT FALSE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (workflow_instance_id) REFERENCES workflow_instance(id)
);
```

#### Stub Tables (For POC)

```sql
-- Simple user table for POC
CREATE TABLE user (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(100) NOT NULL UNIQUE,
    email VARCHAR(255) NOT NULL,
    full_name VARCHAR(255),

    business_object_type VARCHAR(100) NOT NULL,
```

### Service Layer

#### BpmnGenerationService

**Responsibility**: Generate valid Camunda BPMN 2.0 XML from workflow configuration

**Key Methods**:

```
generateBpmn(Long workflowVersionId): String
  Input: Workflow version ID
  Output: Valid BPMN 2.0 XML string
  Logic:
    1. Fetch WorkflowVersion and all WorkflowSteps (ordered by stepOrder)
    2. Fetch all WorkflowStepConditions
    3. Determine workflow structure (sequential, parallel, or mixed)
    4. Build BPMN model:
       - Create BpmnModelInstance
       - Add process element with processDefinitionKey
       - Add start event
       - For each step:
         * If USER_TASK: Create userTask with id=stepKey, name=stepName
         * If END_EVENT: Create endEvent
       - For each condition:
         * If SEQUENTIAL: Create sequenceFlow
         * If PARALLEL_FORK: Create parallelGateway (fork)
         * If PARALLEL_JOIN: Create parallelGateway (join)
       - Connect all elements
    5. Validate BPMN structure
    6. Convert to XML string
    7. Store bpmnXml in WorkflowVersion
  Error Handling: Throw WorkflowException if invalid configuration

deployBpmn(Long workflowVersionId): String
  Input: Workflow version ID
  Output: Camunda deployment ID
  Logic:
    1. Generate BPMN using generateBpmn()
    2. Deploy to Camunda using RepositoryService
    3. Store deploymentId in WorkflowVersion
    4. Update version status to ACTIVE
  Error Handling: Throw WorkflowException if deployment fails
```

**BPMN Generation Algorithm**:

For **Sequential Workflow**:
```xml
<bpmn:process id="purchase-request-v1">
  <bpmn:startEvent id="StartEvent"/>
  <bpmn:sequenceFlow sourceRef="StartEvent" targetRef="DRAFT"/>
  
  <bpmn:userTask id="DRAFT" name="Draft"/>
  <bpmn:sequenceFlow sourceRef="DRAFT" targetRef="SUBMITTED"/>
  
  <bpmn:userTask id="SUBMITTED" name="Submitted"/>
  <bpmn:sequenceFlow sourceRef="SUBMITTED" targetRef="SPONSOR_APPROVED"/>
  
  <!-- ... more tasks ... -->
  
  <bpmn:endEvent id="END"/>
</bpmn:process>
```

For **Parallel Workflow**:
```xml
<bpmn:process id="purchase-request-v2">
  <bpmn:startEvent id="StartEvent"/>
  <bpmn:sequenceFlow sourceRef="StartEvent" targetRef="SUBMITTED"/>
  
  <bpmn:userTask id="SUBMITTED" name="Submitted"/>
  <bpmn:sequenceFlow sourceRef="SUBMITTED" targetRef="ParallelFork"/>
  
  <!-- Fork -->
  <bpmn:parallelGateway id="ParallelFork"/>
  <bpmn:sequenceFlow sourceRef="ParallelFork" targetRef="SPONSOR_APPROVED"/>
  <bpmn:sequenceFlow sourceRef="ParallelFork" targetRef="OWNER_APPROVED"/>
  
  <!-- Parallel tasks -->
  <bpmn:userTask id="SPONSOR_APPROVED" name="Sponsor Approval"/>
  <bpmn:userTask id="OWNER_APPROVED" name="Owner Approval"/>
  
  <bpmn:sequenceFlow sourceRef="SPONSOR_APPROVED" targetRef="ParallelJoin"/>
  <bpmn:sequenceFlow sourceRef="OWNER_APPROVED" targetRef="ParallelJoin"/>
  
  <!-- Join -->
  <bpmn:parallelGateway id="ParallelJoin"/>
  <bpmn:sequenceFlow sourceRef="ParallelJoin" targetRef="MANAGER_APPROVED"/>
  
  <bpmn:userTask id="MANAGER_APPROVED" name="Manager Approval"/>
  <bpmn:sequenceFlow sourceRef="MANAGER_APPROVED" targetRef="END"/>
  
  <bpmn:endEvent id="END"/>
</bpmn:process>
```

**Key Methods**:


### Backend Service Layer

#### AdminService

**Responsibility**: Admin override and management operations

**Key Methods**:

```
completeStepAsAdmin(AdminCompleteRequest): void
  Input: workflowInstanceId, decision (APPROVE/REJECT), reason, adminUserId
  Output: void
  Logic:
    1. Get WorkflowInstance
    2. Get current Camunda task
    3. Complete task with decision
    4. Record in history with isManualOverride=true
    5. Send email notification (if not already sent)
  Error Handling: Throw WorkflowException if task not found
  
moveWorkflowToStep(AdminMoveRequest): void
  Input: workflowInstanceId, targetStepKey, reason, adminUserId
  Output: void
  Logic:
    1. Get WorkflowInstance
    2. Get current and target WorkflowStep
    3. Use Camunda Process Instance Modification API:
       - Cancel current activity
       - Start target activity
    4. Update WorkflowInstance.currentStepKey
    5. Record in history with isManualOverride=true, action=MANUAL_MOVE
  Error Handling: Throw WorkflowException if step invalid or Camunda modification fails
  
cancelWorkflow(AdminCancelRequest): void
  Input: workflowInstanceId, reason, adminUserId
  Output: void
  Logic:
    1. Get WorkflowInstance
    2. Delete Camunda process instance
    3. Update WorkflowInstance.status=CANCELLED
    4. Record in history
  Error Handling: Throw WorkflowException if already completed

getAuditTrail(Long workflowInstanceId): AuditTrailResponse
  Input: Workflow instance ID
  Output: Complete audit trail (history + form submissions + field changes)
  Logic:
    1. Fetch WorkflowInstanceHistory
    2. Fetch FormSubmissions with responses
    3. Fetch FormFieldAudit records
    4. Merge all into timeline structure
    5. Sort by timestamp
  Error Handling: Throw NotFoundException if instance not found
```

## Testing Strategy

### Test Pyramid

```
        /\
       /E2E\         <- Few (optional for POC)
      /------\
     /  API   \      <- Some (integration tests)
    /----------\
   /Unit Tests  \    <- Many (comprehensive unit tests)
  /--------------\
```

### Unit Testing Guidelines

1. **Test in isolation** - Mock all dependencies
2. **Fast execution** - All unit tests should complete in <30s
3. **One assertion per test** - Or closely related assertions
4. **Descriptive names** - test should clearly describe what is being tested
5. **AAA pattern** - Arrange, Act, Assert

Example:
```java
@Test
void shouldThrowExceptionWhenWorkflowVersionNotFound() {
  // Arrange
  when(workflowVersionRepository.findById(999L)).thenReturn(Optional.empty());
  
  // Act & Assert
  assertThrows(WorkflowException.class, () -> bpmnGenerationService.generateBpmn(999L));
}
```

### Integration Testing Guidelines

1. **Test with real Spring context** - Use @SpringBootTest
2. **Use test data** - Load from test-data.sql
3. **Clean between tests** - @Sql to reset state
4. **Test full flows** - From API endpoint through database
5. **Verify Camunda integration** - Actually deploy and execute processes

### Frontend Testing Guidelines

1. **Component tests** - Test each component in isolation
2. **Mock API calls** - Use MSW or similar
3. **Test user interactions** - Simulate clicks, inputs
4. **Test conditional rendering** - Verify show/hide logic
5. **Accessibility** - Test with screen reader queries

### Common Issues and Solutions

**Issue**: H2 database data lost on restart
**Solution**: Expected behavior for in-memory DB. For persistence, switch to file-based: `jdbc:h2:file:./data/workflowdb`

**Issue**: React app not loading
**Solution**: Check if both backend (8080) and frontend (3000) are running. Check proxy configuration in vite.config.ts

**Issue**: BPMN deployment fails
**Solution**: Check BPMN XML validity. Use Camunda Modeler to validate. Check server logs for detailed error

**Issue**: Form validation not working
**Solution**: Check browser console for errors. Verify validation rules in database. Check that field conditions are correctly evaluated

**Issue**: Camunda task not found
**Solution**: Verify process instance exists in Camunda. Check that step key matches user task ID in BPMN

**Issue**: Camunda web interface not accessible (404 error)
**Solution**: 
1. Verify `camunda-bpm-spring-boot-starter-webapp` dependency is in build.gradle
2. Check application logs for Camunda webapp deployment messages
3. Ensure application.yml has `camunda.bpm.webapp.index-redirect-enabled: true`
4. Access full path: http://localhost:8080/camunda/app/cockpit/default/
5. Check if other endpoints work (Swagger, H2 console) to verify application started correctly

**Issue**: Cannot login to Camunda (admin/admin not working)
**Solution**: 
1. Verify `camunda.bpm.admin-user.id` and `password` in application.yml
2. Check logs for user creation messages
3. Try accessing H2 console and check `ACT_ID_USER` table for admin user
4. If user doesn't exist, Camunda may not have initialized properly - check startup logs

**Issue**: Process definition not visible in Camunda Cockpit
**Solution**:
1. Verify BPMN was deployed successfully (check logs)
2. Check `ACT_RE_PROCDEF` table in H2 console
3. Verify deployment ID stored in workflow_version table
4. Try redeploying via API endpoint
5. Check BPMN XML is valid using Camunda Modeler

**Issue**: Tasks not appearing in Camunda Tasklist
**Solution**:
1. Verify process instance is running (check Cockpit)
2. Verify user task exists in current process execution (Cockpit → Process Instances → Activity Instance Tree)
3. Check task assignee/candidate configuration
4. Query `ACT_RU_TASK` table in H2 console to see active tasks
5. Verify task assignment logic in workflow service

### Appendix E: BPMN Generation Examples

**Example 1: Simple Sequential Workflow**
```
Input (Database):
  Steps: DRAFT → SUBMITTED → APPROVED → END
  Conditions: All SEQUENTIAL

Output (BPMN):
  StartEvent → UserTask(DRAFT) → UserTask(SUBMITTED) → UserTask(APPROVED) → EndEvent
```

**Example 2: Parallel Workflow**
```
Input (Database):
  Steps: SUBMITTED → SPONSOR_APPROVED, OWNER_APPROVED (parallel) → MANAGER_APPROVED → END
  Conditions: 
    - SUBMITTED → SPONSOR_APPROVED (PARALLEL_FORK)
    - SUBMITTED → OWNER_APPROVED (PARALLEL_FORK)
    - SPONSOR_APPROVED → MANAGER_APPROVED (PARALLEL_JOIN)
    - OWNER_APPROVED → MANAGER_APPROVED (PARALLEL_JOIN)

Output (BPMN):
  StartEvent → UserTask(SUBMITTED) → ParallelGateway(Fork) 
    → [UserTask(SPONSOR_APPROVED), UserTask(OWNER_APPROVED)] 
    → ParallelGateway(Join) → UserTask(MANAGER_APPROVED) → EndEvent
```

## Final Notes for Devin

### Code Quality Standards

1. **Follow Java naming conventions**
   - Classes: PascalCase
   - Methods/variables: camelCase
   - Constants: UPPER_SNAKE_CASE

2. **Use Lombok annotations**
   - @Data for simple DTOs
   - @Builder for complex objects
   - @Slf4j for logging

3. **Use MapStruct for DTO mapping**
   - Don't write manual mapping code
   - Define @Mapper interfaces

4. **Write meaningful commit messages**
   - "feat: Add dynamic form rendering"
   - "fix: Correct validation for date fields"
   - "test: Add integration tests for workflow service"

5. **Add JavaDoc for public methods**
   - Especially service layer methods
   - Explain parameters and return values

6. **Use constants instead of magic numbers/strings**
   ```java
   public class WorkflowConstants {
     public static final String STATUS_IN_PROGRESS = "IN_PROGRESS";
     public static final int MAX_RETRY_ATTEMPTS = 3;
   }
   ```

### Testing Best Practices

1. **Test happy path AND edge cases**
2. **Use descriptive test names** - testShouldThrowExceptionWhenWorkflowNotFound()
3. **Don't test getters/setters** - Focus on business logic
4. **Use @BeforeEach for test setup** - Keep tests DRY
5. **Assert one thing per test** - Or closely related things
6. **Use test data builders** for complex objects

### Common Pitfalls to Avoid

1. **DON'T expose entities in REST APIs** - Always use DTOs
2. **DON'T catch generic Exception** - Catch specific exceptions
3. **DON'T ignore validation** - Validate all input
4. **DON'T hardcode values** - Use configuration
5. **DON'T skip transaction management** - Use @Transactional appropriately
6. **DON'T forget to close resources** - Use try-with-resources
7. **DON'T log sensitive data** - Passwords, tokens, PII
