## 1. Executive Summary

### Purpose
Build a new **Data Integration Service** that exposes workflow application data to external partner systems through multiple consumption channels (Search, Bulk, Events) with a unified response structure.

### Key Requirements
- **Search API**: Elasticsearch-backed search returning summary data (~60% of fields)
- **Bulk API**: Paginated batch data access for partner synchronization (2-3 times per day)
- **Event Push**: Real-time workflow state change notifications via webhooks
- **AI Agent API**: Detailed data including all descriptive fields (~100% of fields)

### Core Design Principles
1. **Unified Response Structure**: Same JSON shape across all APIs (domains nullable based on partner config)
2. **Configuration-Driven**: Partner onboarding via config, not code changes
3. **Performance Isolation**: Bulk operations don't impact Search/Events (separate thread pools)
4. **Reliability**: Comprehensive error handling, retry mechanisms, dead letter queues
5. **View-Based Data Access**: Query database views instead of joining entities

### Integration Pattern
Core Workflow Application publishes Spring Events when workflow state changes. An async event listener in the Core App makes a REST API call to the Data Integration Service, which then processes the event and notifies partner systems via configured webhooks.

## 2. System Architecture

### High-Level Architecture

**Component Overview:**
- **Core Workflow Application**: Existing application managing workflows, publishes events on state changes
- **Data Integration Service**: NEW microservice that exposes data to partners
- **Database Views**: Read-only views (summary and detailed) in Core App database
- **Elasticsearch**: Search index for quick ID lookup
- **Partner Systems**: External systems consuming data via APIs/webhooks

**Data Flow:**

**Event Flow:**
1. Core Workflow App: State change occurs
2. Core App: Spring Event published (within transaction)
3. Core App: Async listener catches event
4. Core App: REST call to Integration Service (with retry logic)
5. Integration Service: Receives event, queues for processing
6. Integration Service: Queries database view for application data
7. Integration Service: Filters data per partner configuration
8. Integration Service: Calls partner webhooks in parallel
9. Integration Service: Logs delivery status, handles failures

**Search Flow:**
1. Partner: Calls Search API with ~20 search parameters
2. Integration Service: Queries Elasticsearch with search params
3. Elasticsearch: Returns matching application IDs
4. Integration Service: Fetches application data from summary view
5. Integration Service: Filters data per partner configuration
6. Integration Service: Returns JSON response

**Bulk Flow:**
1. Partner: Calls Bulk IDs API with "since" timestamp
2. Integration Service: Queries summary view for IDs modified since timestamp
3. Integration Service: Returns list of (ID, lastModified)
4. Partner: Calls Bulk Data API with subset of IDs (paginated)
5. Integration Service: Fetches data from summary view
6. Integration Service: Filters per partner config, returns page

### Architectural Decisions

**Why Separate Service?**
- Core App focuses on workflow management
- Integration Service handles data distribution concerns
- Independent scaling and deployment
- Core App performance unaffected by partner queries

**Why Database Views?**
- Pre-joined data (no complex joins in application)
- Database optimizer handles query planning
- Consistent data composition
- Two views: summary (60% fields) and detailed (100% fields)
- 75% of traffic uses lightweight summary view

**Why Configuration-Driven?**
- Add new partners without code deployment
- Change partner data requirements independently
- Clear audit trail of configuration changes
- Easy to test and validate partner-specific behavior

## 3. Technical Stack

### Framework & Core Libraries
- **Spring Boot 3.2.x**: Core framework
- **Java 17 or 21**: Programming language
- **Maven 3.9.x**: Build tool
- **Spring Data JPA**: Database access (single datasource)
- **HikariCP**: Connection pooling (included in Spring Boot)
- **MapStruct 1.5.x**: DTO mapping (compile-time, no reflection)
- **Resilience4j 2.1.x**: Circuit breaker and retry 

### Data & Search
- **Oracle**: Database (same as Core App)
- **Spring Data Elasticsearch 5.2.x**: Search integration


### Utilities & Monitoring
- **Lombok 1.18.x**: Reduce boilerplate
- **Google Guava 32.x**: Rate limiting utilities (in-memory, single-pod)
- **Jackson**: JSON serialization (included in Spring Boot)
- **Spring Boot Actuator**: Metrics and health checks
- **Micrometer**: Metrics collection (need analysis)
- **Prometheus**: Metrics export format (need analysis)

### Testing
- **JUnit 5**: Unit testing
- **Mockito**: Mocking framework
- **Spring Boot Test**: Integration testing


### Documentation
- **SpringDoc OpenAPI 2.x**: API documentation (Swagger UI)

### Why These Choices

**Resilience4j over Custom Code**:
- Circuit breaker and retry are complex to implement correctly
- Resilience4j is proven, tested, and provides excellent metrics
- Configuration-based (no custom state management code needed)


**MapStruct over ModelMapper**:
- Compile-time code generation (no reflection overhead)
- Type-safe mapping
- Clear error messages at compile time
- Better performance

**Single Datasource**:
- Same database = single pool is natural design
- No entity manager complexity
- No transaction coordination issues
- Thread isolation provides workload separation

**Guava for Rate Limiting**: (needs analysis)
- Simple, in-memory, works well for single pod
- For multi-pod: Upgrade to Redis-based rate limiting (future enhancement)

### Single-Pod Assumptions (Documented)

The MVP makes these assumptions for simplicity:
- **Single pod deployment**: Rate limiting, circuit breaker state, ES sync checkpoint are in-memory: NOTE: For MVP we need at least two pods. Cannot have redis. Since it is served internally, we can avoid rate limiting.
- **Low traffic**: 50 records/day workload doesn't require distributed coordination (NEED COUNT FROM PROD)
- **Future scaling**: When deploying multiple pods, upgrade to Redis for shared state

These are acceptable trade-offs for MVP that reduce complexity significantly.

### Infrastructure Requirements
- **Runtime**: OpenJDK 17/21, 2-4GB heap, 2-4 CPU cores
- **Database**: Read-only access to Core App database views
- **Network**: Access to Core App API and partner webhook endpoints

### Integration Service Tables

#### partner_master (NEEDS REVIEW)
**Purpose**: Store operational partner configuration
**Key Fields**:
- partner_id (PK): Unique identifier (e.g., "partner-a")
- partner_name: Display name
- enabled: Master kill switch (true/false)
- webhook_url: Partner's webhook endpoint for events
- webhook_auth_type: BEARER_TOKEN, BASIC_AUTH, API_KEY, NONE
- webhook_auth_token: Encrypted auth token
- webhook_timeout_seconds: Default 30
- circuit_breaker_enabled: true/false
- circuit_breaker_threshold: Default 10 consecutive failures
- circuit_breaker_cooldown_minutes: Default 15
- priority: HIGH, MEDIUM, LOW (affects retry behavior)
- contact_email: Partner technical contact
- Audit fields: created_at, created_by, updated_at, updated_by

**Indexes**: partner_id (PK), enabled

#### partner_config_audit (NEEDS REVIEW)
**Purpose**: Track all configuration changes for audit trail
**Key Fields**:
- id (PK): Auto-increment
- partner_id: Reference to partner_master
- config_version: Version number from YAML file
- changed_by: User who made change
- changed_at: Timestamp
- change_type: DOMAIN_ADDED, DOMAIN_REMOVED, WEBHOOK_CHANGED, ENABLED, DISABLED
- previous_value: JSON snapshot before change
- new_value: JSON snapshot after change
- change_reason: Free text explanation
- approved_by: Approver (if approval workflow exists)

**Indexes**: (partner_id, changed_at)

#### event_delivery_log (NEEDS REVIEW)
**Purpose**: Log every webhook delivery attempt for observability
**Key Fields**:
- id (PK): Auto-increment
- event_id: UUID from Core App (for correlation)
- partner_id: Which partner
- application_id: Which application
- event_type: WORKFLOW_STATE_CHANGED
- payload_size_bytes: Response size
- attempt_number: 1, 2, 3, etc.
- attempt_timestamp: When attempt was made
- http_status_code: Response code from partner
- response_body: Truncated response (first 1000 chars)
- response_time_ms: Latency
- delivery_status: SUCCESS, FAILED, RETRYING
- error_message: Exception message if failed
- next_retry_at: When to retry if failed

**Indexes**: event_id, (partner_id, attempt_timestamp), (delivery_status, next_retry_at)



#### partner_event_failures (NEEDS REVIEW)
**Purpose**: Dead letter queue for events that failed all retries
**Key Fields**:
- id (PK): Auto-increment
- event_id: UUID (unique)
- partner_id: Which partner couldn't receive it
- application_id: Which application
- event_payload: Full JSON payload for replay
- total_attempts: How many times we tried
- first_attempt_at: When first tried
- last_attempt_at: When last tried
- failure_category: PARTNER_DOWN, PARTNER_REJECTED, NETWORK_ERROR, TIMEOUT
- permanent_failure_reason: Why we gave up
- status: PENDING_RETRY, PERMANENTLY_FAILED, MANUALLY_RESOLVED
- resolved_at: When manually fixed
- resolved_by: Who fixed it

**Indexes**: (status, last_attempt_at), (partner_id, status)



#### event_propagation_failures (In Core App Database) (NEEDS REVIEW)
**Purpose**: Dead letter queue in Core App for events that couldn't reach Integration Service
**Key Fields**:
- id (PK): Auto-increment
- event_id: UUID
- application_id: Which application
- event_type: WORKFLOW_STATE_CHANGED
- event_payload: Full event data
- failure_reason: Why REST call failed
- http_status_code: If got response
- retry_count: How many retries attempted
- first_failed_at: When first failed
- last_retry_at: When last retry attempted
- status: PENDING_RETRY, PERMANENTLY_FAILED, RESOLVED
- resolved_at: When successfully sent

### Module Dependency Rules

**Dependency Flow** (modules can only depend on layers below):
```
┌─────────────────────────────────────────────┐
│  API Modules (search, bulk, events, admin) │  ← Feature modules
└──────────────────┬──────────────────────────┘
                   ↓
┌──────────────────▼──────────────────────────┐
│  Shared Module (projection, configuration)  │  ← Business logic
└──────────────────┬──────────────────────────┘
                   ↓
┌──────────────────▼──────────────────────────┐
│  Common Module (exceptions, security, utils)│  ← Infrastructure
└─────────────────────────────────────────────┘
```

**Allowed Dependencies**:
- ✅ Search module → Shared module (uses ProjectionService)
- ✅ Bulk module → Shared module (uses ProjectionService)
- ✅ Events module → Shared module (uses ProjectionService, ConfigurationService)
- ✅ All modules → Common module (exceptions, utilities)
- ✅ Shared module → Common module

**Forbidden Dependencies**:
- ❌ Search module → Bulk module (modules are independent)
- ❌ Bulk module → Events module (no cross-module dependencies)
- ❌ Shared module → Any API module (shared is lower layer)
- ❌ Common module → Any other module (common is lowest layer)

### Module Characteristics

**SHARED MODULE** (Core Business Logic):
- **Purpose**: Reusable business logic used by all API modules
- **Components**: ProjectionService, ConfigurationService, view repositories
- **Owns**: Data projection logic, partner configuration management
- **Independence**: Can be unit tested without any API module
- **Future**: Could become a library if services split

**SEARCH MODULE** (Search API Feature):
- **Purpose**: Application search functionality
- **API**: POST /api/v1/search
- **Owns**: Elasticsearch integration, search query building
- **Dependencies**: Shared (ProjectionService, ConfigurationService)
- **Thread Pool**: Uses default Tomcat pool
- **Future**: Could be extracted as Search Service

**BULK MODULE** (Bulk Sync Feature):
- **Purpose**: Batch data synchronization
- **API**: GET /api/v1/bulk/ids, GET /api/v1/bulk/data
- **Owns**: Bulk data retrieval, pagination logic
- **Dependencies**: Shared (ProjectionService, ConfigurationService)
- **Thread Pool**: Dedicated bulk executor (isolated)
- **Future**: Could be extracted as Bulk Service

**EVENTS MODULE** (Event Processing Feature):
- **Purpose**: Real-time event distribution to partners AND AI agents
- **API**: POST /api/v1/events/workflow-state-changed
- **Owns**: Event ingestion, webhook delivery, retry logic, dead letter queue
- **Consumers**: 
  - Partner systems (push via webhooks, uses summary projection)
  - AI agents (push via webhooks, uses detailed projection with descriptive fields)
- **Dependencies**: Shared (ProjectionService, ConfigurationService)
- **Thread Pools**: Event executor + webhook executor (isolated)
- **Key Difference**: Partners get summary data (60% fields), AI agents get detailed data (100% fields including comments, descriptions, notes)
- **Future**: Could be extracted as Event Service

**ADMIN MODULE** (Operations Interface):
- **Purpose**: Operational management and troubleshooting
- **API**: POST /admin/partners/{id}/enable, /disable, /test, /retry
- **Owns**: Partner management, failure investigation
- **Dependencies**: Shared (ConfigurationService), Events (EventPublishingService for test)
- **Thread Pool**: Uses default Tomcat pool
- **Future**: Could be web UI instead of API

**CONFIGURATION MODULE** (Partner Config Management):
- **Purpose**: Partner configuration loading and management
- **Owns**: YAML loading, database config, merging logic
- **Dependencies**: Common (utilities)
- **Used By**: Shared module (ConfigurationService)

**COMMON MODULE** (Infrastructure):
- **Purpose**: Cross-cutting concerns
- **Owns**: Exception handling, security, metrics, utilities
- **Dependencies**: None (lowest layer)
- **Used By**: All modules

### Benefits of Modular Monolith Design

**1. Clear Boundaries**:
- Each module has single responsibility
- Easy to understand what each module does
- New developers can focus on one module at a time

**2. Independent Development**:
- Teams can work on different modules in parallel
- Search changes don't affect Bulk module
- Events changes don't affect Search module

**3. Testability**:
- Unit test modules independently
- Mock dependencies between modules
- Integration test per module or end-to-end

**4. Future Microservices Path**:
- Each module is already isolated
- Clear APIs between modules (service interfaces)
- Can extract module to separate service if needed
- Example: High event volume → Extract Events module as separate service

**5. Deployment Simplicity**:
- Single JAR to deploy (all modules together)
- No distributed system complexity (yet)
- Simple troubleshooting (single log file)
- No network calls between modules (in-process)

**6. Resource Sharing**:
- Single database connection pool (all modules share)
- Single Elasticsearch connection
- Shared configuration and utilities
- Lower operational overhead

### When to Split Modules into Microservices

Keep as modular monolith unless:

**Volume-Based**:
- Events: More events getting generated
- Search: Search volume grows
- Bulk: Daily volume impacts other APIs

**Team-Based**:
- Separate teams own Search vs Events
- Different deployment schedules needed
- Different technology requirements

**Scaling-Based**:
- Events needs 10 pods, Search needs 3 pods
- Current: Scale entire monolith (waste resources)
- After split: Scale each service independently

**For 50 records/day workload**: Modular monolith is perfect. All modules in one service, clear boundaries, simple operations. (FIND THE RIGHT NUMBER FROM PROD DB)

### Key Services (Shared Module)

#### ProjectionService
**Responsibility**: Build and filter data projections from database views

**Key Methods**:
- Build summary projection (single application)
- Build detailed projection (single application)
- Build summary projections in batch (for bulk API)
- Filter projection based on partner configuration
- Apply domain-level and field-level filtering

**Data Sources**:
- Queries: application_data_summary_view (primary)
- Queries: application_data_detailed_view (AI Agent only)
- Uses: MapStruct for view entity → DTO mapping
- Caching: Consider caching frequently accessed applications (optional)

**Key Logic**:
- Load data from appropriate view based on channel (summary vs detailed)
- Map view entity to ApplicationDataProjection DTO
- Apply partner-specific domain filtering (set excluded domains to null)
- Apply field-level exclusions if configured
- Return unified structure (fixed JSON shape)

#### ConfigurationService
**Responsibility**: Manage partner configurations (merge YAML + database)

**Key Methods**:
- Get merged partner configuration by ID
- Get channel-specific configuration (resolves overrides)
- Check if partner is enabled (quick boolean check)
- Get included domains for partner + channel
- Refresh configuration from sources (cache invalidation)
- Get all active partners (for event broadcasting)

**Configuration Sources**:
- YAML files: resources/partner-configs/{partnerId}.yaml (structural config)
- Database: partner_master table (operational config)
- Merge strategy: Database overrides YAML for operational fields (enabled, webhook_url, auth_token)
- Caching: In-memory cache with 5-minute TTL, manual refresh endpoint

**Configuration Resolution**:
- Start with system defaults
- Apply partner default configuration from YAML
- Apply channel-specific overrides if present
- Merge database operational settings
- Result: Fully resolved configuration for partner + channel

#### EventProcessingService
**Responsibility**: Process workflow events from Core App

**Key Methods**:
- Process workflow state change event (main entry point, async)
- Determine interested partners for event type
- Build projection for each interested partner
- Coordinate event publishing to partners

**Processing Flow**:
1. Receive event from Event Ingestion API
2. Mark event as received (idempotency check)
3. Query all partners with events enabled
4. For each partner:
   - Check if partner is enabled
   - Check circuit breaker status
   - Build projection from summary view
   - Filter by partner configuration
   - Queue for webhook delivery
5. Return 202 Accepted to Core App (don't block)
6. Process asynchronously

**Error Handling**:
- If projection fails: Log error, skip partner, continue with others
- If all partners fail: Still return success to Core App (events stored for retry)
- Idempotency: Track processed event_ids, ignore duplicates

#### EventPublishingService
**Responsibility**: Deliver events to partner webhooks with retry logic

**Key Methods**:
- Publish to partner webhook (with retries)
- Check circuit breaker status
- Open/close circuit breaker manually
- Log delivery attempts
- Handle failure scenarios

**Delivery Flow**:
1. Check if circuit breaker is open (skip if open)
2. Build HTTP request (headers, auth, payload)
3. Call partner webhook endpoint
4. Log delivery attempt (success/failure)
5. If success: Update metrics, done
6. If failure:
   - Determine if retriable (5xx, timeout) or not (4xx)
   - If retriable: Schedule retry with exponential backoff
   - If not retriable: Log as permanent failure, alert
   - Update circuit breaker failure count
   - If threshold reached: Open circuit

**Retry Strategy**:
- Immediate: 3 attempts with 5s, 10s, 20s backoff
- Short-term: Every 15 minutes for 2 hours
- Long-term: Every 1 hour for 24 hours
- After 24 hours: Move to dead letter queue
- Total: ~30 attempts over 24 hours

**Circuit Breaker Logic**:
- Closed (normal): Deliver events normally
- Open (broken): Skip delivery, save to dead letter, don't attempt
- Half-Open (testing): After cooldown, try 1 event
  - If succeeds: Close circuit, resume normal delivery
  - If fails: Open circuit again

#### ElasticsearchSyncService
**Responsibility**: Keep Elasticsearch index in sync with database

**Key Methods**:
- Scheduled incremental sync (every 5 minutes)
- Full reindex (manual trigger)
- Sync single application (called from event processing)

**Sync Strategy**:
- Track last sync timestamp
- Query summary view for applications modified since last sync
- Index only searchable fields (~20 fields) in Elasticsearch
- Update last sync timestamp
- Log sync statistics (records synced, errors, duration)

**Indexed Fields** (subset of summary view):
- Application metadata: id, reference_number, workflow_state, priority, created_date
- Contact: primary_email, primary_phone, contact_person_name

## 6. API Specifications

### API Naming Convention
- Base path: `/api/v1`
- Versioning: URL-based (v1, v2, etc.)
- Response format: JSON
- Date format: ISO 8601 (yyyy-MM-dd'T'HH:mm:ss'Z')

### Design Pattern: RESTful Manifest Pattern

The Bulk API implements a **manifest-first pattern** (also known as a control file pattern in traditional batch systems) adapted for REST APIs:

**Traditional Control File Approach:**
1. Generate control file (contains record count, checksums, metadata)
2. Generate data file (contains actual records)
3. Partner downloads control file first
4. Partner downloads data file
5. Partner validates using control file metadata

**Our RESTful Manifest Pattern:**
1. IDs Endpoint = "Manifest" (contains totalElements, list of IDs)
2. Data Endpoint = "Data retrieval" (contains actual records)
3. Partner calls IDs endpoint first
4. Partner calls Data endpoint for batches
5. Partner validates using totalElements from manifest

**Benefits of RESTful Approach:**
- ✅ No file management (FTP, storage, cleanup)
- ✅ Real-time data (not batch windows)
- ✅ Partial sync recovery (retry specific IDs)
- ✅ Standard HTTP error handling
- ✅ API-based authentication (simpler than SFTP keys)
- ✅ Flexible batching (partner controls batch size)

**What We Provide (Like Control File Metadata):**
- Total record count (`totalElements`)
- Record identifiers with timestamps
- Pagination information
- Standard response structure

**What Partner Does (Like Control File Validation):**
- Compares expected count vs received count
- Identifies missing records (if any)
- Retries missing records
- Maintains sync state (last sync timestamp)

This pattern provides the reliability and completeness guarantees of traditional control files while leveraging modern REST API benefits.

### 6.3 Bulk API - IDs Endpoint

**GET /api/v1/bulk/ids**

**Purpose**: Get list of application IDs and timestamps modified since given time. This endpoint provides a "manifest" of available records that partners use to validate completeness of their sync.

**Request Headers**:
- Authorization: Bearer {partner_token}
- X-Partner-ID: partner-a

**Query Parameters**:
- since: ISO 8601 timestamp (required)
- page: Page number (default: 0)
- size: Page size (default: 1000, max: 5000)

**Example**: `/api/v1/bulk/ids?since=2024-11-01T00:00:00Z&page=0&size=1000`

**Response**: 200 OK
```
{
  "content": [
    {
      "applicationId": 12345,
      "lastModified": "2024-11-01T10:30:00Z"
    },
    {
      "applicationId": 12346,
      "lastModified": "2024-11-01T11:45:00Z"
    }
    // ... up to 1000 records per page
  ],
  "page": 0,
  "size": 1000,
  "totalElements": 5000,  // ← CRITICAL: Total count for partner validation
  "totalPages": 5
}
```

**Partner Validation Pattern**:

Partners use `totalElements` to validate sync completeness:

1. **Fetch All IDs**: Call this endpoint for all pages to get complete list
2. **Store Locally**: Save all IDs from manifest (5000 in this example)
3. **Fetch Data**: Call Bulk Data API for batches of IDs
4. **Validate Completeness**: 
   - Count records stored in partner database
   - Compare: `totalElements` (5000) vs records stored (should be 5000)
   - If mismatch: Identify missing IDs (set difference) and retry
5. **Recovery**: Re-fetch specific missing IDs via Bulk Data API

**Implementation Note**:
The `totalElements` field is automatically provided by Spring Data Pagination and represents the total count of records modified since the `since` timestamp. Partners should use this for reconciliation - it serves the same purpose as a "record count" in traditional control file patterns.

**Why This Design**:
- Simple manifest-first pattern (similar to control files)
- Partners validate on their side (clear responsibility boundary)
- Idempotent (can re-query manifest anytime)
- Supports partial sync recovery (retry specific missing IDs)

**Error Responses**:
- 400 Bad Request: Invalid since parameter
- 401 Unauthorized: Invalid partner token
- 403 Forbidden: Partner not enabled or no bulk access
- 429 Too Many Requests: Rate limit exceeded (max 1 concurrent request per partner)



### 6.4 Bulk API - Data Endpoint

**GET /api/v1/bulk/data**

**Purpose**: Get full application data for specified IDs (paginated). Partners call this after retrieving IDs from the IDs endpoint.

**Request Headers**:
- Authorization: Bearer {partner_token}
- X-Partner-ID: partner-a

**Query Parameters**:
- ids: Comma-separated list of application IDs (required, max 1000 per request)
- page: Page number (default: 0)
- size: Page size (default: 100, max: 1000)

**Example**: `/api/v1/bulk/data?ids=12345,12346,12347&page=0&size=100`

**Response**: 200 OK (same structure as Search API response)
```
{
  "content": [
    {
      "applicationId": 12345,
      "lastModified": "2024-11-01T10:30:00Z",
      "workflowState": "APPROVED",
      "baseInfo": { ... },
      "contactPoints": { ... },
      "questionnaire": null,  // Excluded for this partner
      "assessment": { ... },
      "meetings": { ... },
      "issues": null,  // Excluded for this partner
      "postCompletionReview": null  // Excluded for this partner
    }
    // ... more records
  ],
  "page": 0,
  "size": 100,
  "totalElements": 1000,
  "totalPages": 10
}
```

**Response Notes**:
- Fixed structure: All domain keys present
- Excluded domains: Set to null based on partner configuration
- Pagination: Standard Spring Data pagination format
- Same projection filtering as Search API (per partner config)

**Typical Usage Pattern**:
```
1. Partner calls /bulk/ids → Gets 5000 IDs (manifest)
2. Partner batches IDs into groups of 100-500
3. For each batch: Call /bulk/data?ids={batch}
4. Store received data in partner database
5. After all batches: Validate count (5000 expected vs stored)
```

**Performance Considerations**:
- Recommend 100-500 IDs per request (balance between API calls and response size)
- Don't send all 5000 IDs in one request (timeout risk, large payload)
- Process batches sequentially or with limited parallelism (respect rate limits)

**Error Responses**:
- 400 Bad Request: No IDs provided or too many IDs (>1000)
- 401 Unauthorized: Invalid partner token
- 403 Forbidden: Partner not enabled or no bulk access
- 429 Too Many Requests: Rate limit exceeded (max 2 concurrent requests per partner)
- 503 Service Unavailable: Circuit breaker open (bulk service overloaded)


**Rate Limiting**:
- Per partner: Max 2 concurrent requests
- Global: Max 10 concurrent requests
- If limit exceeded: Return 429 with Retry-After header

**Reconciliation Note**:
Partners should track which IDs they've successfully fetched and stored. If sync is interrupted, partners can resume by requesting only the remaining IDs from their manifest, not re-fetching the entire dataset.

## 8. Error Handling Strategy

### Error Categories & Handling

**Layer 1: Core App → Integration Service**

**Error Scenarios**:
- Integration Service is down (connection refused)
- Integration Service is slow (timeout)
- Integration Service returns 500
- Network issues

**Handling in Core App**:
- Immediate retry: 3 attempts with exponential backoff (2s, 4s, 8s)
- Total time: ~14 seconds maximum
- If all retries fail: Store in event_propagation_failures table
- Async listener ensures workflow is not blocked
- Scheduled job retries from dead letter every 15 minutes
- After 24 hours or 10 attempts: Mark as PERMANENTLY_FAILED
- Alert if failures > 10 in 1 hour (indicates Integration Service down)

**Recovery**:
- Automatic: Scheduled retry job
- Manual: Admin dashboard to replay failed events
- Monitoring: Track dead letter queue size

**Layer 2: Integration Service → Partner Webhooks**

**Error Scenarios**:
- Partner webhook is down (5xx, connection refused, timeout)
- Partner rejects payload (4xx)
- Partner authentication fails (401)
- Partner rate limits us (429)

**Retriable Errors** (Network, 5xx, Timeouts, 429):
- Immediate: 3 attempts with 5s, 10s, 20s backoff
- Short-term: Every 15 minutes for 2 hours
- Long-term: Every 1 hour for 24 hours
- Total: ~30 attempts over 24 hours

**Non-Retriable Errors** (4xx except 429):
- 400 Bad Request: Log error, alert team (payload issue), don't retry
- 401 Unauthorized: Alert immediately (auth token invalid), open circuit, don't retry
- 404 Not Found: Alert (webhook URL wrong), open circuit, don't retry
- 403 Forbidden: Alert (permission issue), don't retry

**Circuit Breaker Logic**:
- Track consecutive failures per partner
- Threshold: 10 consecutive failures
- Action: Open circuit for 15 minutes
- Effect: Stop attempting deliveries, save all events to dead letter
- After cooldown: Try 1 event (half-open state)
  - Success: Close circuit, resume normal delivery
  - Failure: Open circuit for another 15 minutes

**Dead Letter Queue Management**:
- Store in partner_event_failures table
- Include full payload for replay
- Track failure category and reason
- Retention: Keep until manually resolved or 30 days
- Manual replay: Admin dashboard with filters and bulk actions

### Error Notification Strategy

**Critical Alerts** 
- Integration Service is down (5+ consecutive failures from Core App in 5 minutes)
- Partner authentication failure (401 from any partner)
- Systematic failures (50+ events failed to ANY partner in 1 hour)
- Core App dead letter growing rapidly (100+ events in 1 hour)

**Warning Alerts** 
- Single partner issues (10+ failures to specific partner in 1 hour)
- Slow partner performance (response time > 10 seconds)
- Unprocessed dead letters older than 1 hour
- Circuit breaker opened for partner

**Daily Reports** (Email Summary):
- Total events processed
- Successful deliveries by partner
- Failed deliveries by partner
- Average delivery time per partner
- Dead letter queue sizes
- Top error messages

### Error Metadata Capture

**For Core App → Integration Service Failures**:
- Event ID, Application ID, Event type, Timestamp
- Full event payload
- HTTP status code (if received)
- Response body (if any)
- Exception stack trace
- Network error details
- Core App server hostname
- Retry attempt number

**For Integration Service → Partner Failures**:
- All above, plus:
- Partner ID
- Partner webhook URL
- Request headers sent (excluding auth tokens)
- Response headers received
- Response body (first 1000 characters)
- Which domains were included
- Payload size in bytes
- Integration Service instance ID
- Circuit breaker state before/after

### Error Prevention Strategies

**Payload Validation**:
- Validate payload against JSON schema before sending
- Check all required fields present
- Check field types match expected
- If validation fails: Log as bug, alert team, don't send

**Partner Health Checks**:
- Scheduled job every 5 minutes
- Send test event (special event type partners should ignore)
- Monitor: Is webhook reachable? Auth working? Response time acceptable?
- Pre-emptively open circuit breaker if health check fails

**Contract Testing**:
- During partner onboarding
- Share JSON schema with partner
- Send sample payloads to partner test environment
- Partner confirms they can parse correctly
- Document agreed-upon structure

**Graceful Degradation**:
- If Integration Service is struggling (high load):
  - Prioritize high-priority partners
  - Queue low-priority events for later processing
  - Reduce retry frequency
  - Alert operations team
  - Don't fail workflow in Core App

### Error Handling Decision Matrix

| Error Scenario | Immediate Action | Retry Strategy | Alert Level | Resolution Owner |
|----------------|------------------|----------------|-------------|------------------|
| Integration Service down | Dead letter in Core App | Auto-retry 15min | Critical | DevOps |
| Integration Service slow (timeout) | Retry with backoff | 3 attempts | Warning | DevOps |
| Partner webhook down | Dead letter in Integration Service | Exponential 24hr | Warning | Partner + Ops |
| Partner returns 401 | Open circuit, don't retry | None | Critical | Ops (check credentials) |
| Partner returns 400 | Log and alert | None | Critical | Dev (payload issue) |
| Partner returns 500 | Retry with backoff | Exponential 24hr | Info (if occasional) | Partner |
| Partner timeout | Retry with backoff | Exponential 24hr | Warning (if frequent) | Partner + Ops |
| Payload validation fails | Log, alert, don't send | None | Critical | Dev (bug in code) |
| Elasticsearch sync fails | Log and retry | Next sync cycle | Warning | DevOps |

## 9. Performance & Isolation

### Performance Requirements

**API Response Time SLAs**:
- Search API: P50 < 100ms, P95 < 500ms, P99 < 1s
- Bulk IDs API: P50 < 1s, P95 < 2s
- Bulk Data API: P50 < 5s, P95 < 10s (per 1000 records)
- Event Processing: P50 < 200ms, P95 < 5min, P99 < 10min (end-to-end)

**Resource Utilization Targets**:
- Thread Pools: Normal < 50%, Peak < 80%, Alert > 90% for 5 minutes
- Connection Pools: Normal < 60%, Peak < 80%, Alert > 90% for 5 minutes
- CPU: Normal < 50%, Peak < 70%, Alert > 85% for 5 minutes
- Memory (Heap): Normal < 60%, Peak < 75%, Alert > 85%

### Thread Pool Isolation Strategy

**Why Needed**: Bulk API operations (though infrequent) should not block Search API or Event processing

**Thread Pool Configuration**:

**Default Thread Pool** (Tomcat):
- Size: 200 threads
- Handles: Search API, Event Ingestion API, Admin API
- Purpose: Quick operations, low latency

**Bulk Thread Pool** (Dedicated):
- Core threads: 5
- Max threads: 10
- Queue capacity: 10
- Thread name prefix: "bulk-"
- Handles: Bulk API only (both IDs and Data endpoints)
- Purpose: Isolate potentially long-running bulk operations

**Event Processing Thread Pool** (Dedicated):
- Core threads: 10
- Max threads: 20
- Queue capacity: 100
- Thread name prefix: "event-"
- Handles: Async event processing after ingestion
- Purpose: Dedicated capacity for event processing

**Webhook Delivery Thread Pool** (Dedicated):
- Core threads: 15
- Max threads: 30
- Queue capacity: 200
- Thread name prefix: "webhook-"
- Handles: Partner webhook calls
- Purpose: I/O bound operations, many concurrent requests

**Benefits**:
- Bulk operations can't exhaust all threads
- Search API always has threads available
- Events processing not blocked by bulk
- Clear separation of concerns

### Connection Pool Strategy

**Single Shared Connection Pool** (ONLY APPROACH):
- Maximum pool size: 30 connections
- Minimum idle: 10 connections
- Connection timeout: 10 seconds
- Idle timeout: 10 minutes
- Max lifetime: 30 minutes
- Leak detection threshold: 60 seconds

**Why Single Pool is the Right Choice**:
- Same database, same service = single pool is natural design
- Thread pool isolation already provides workload separation
- Bulk limited to 10 threads = max 10 connections used simultaneously
- Search/Events have 20+ connections always available
- Simple architecture, easy to maintain and debug

**IMPORTANT - Do NOT Use Multiple Connection Pools**:
Multiple connection pools pointing to the same database adds massive complexity:
- Requires separate EntityManager configurations
- Creates transaction management nightmares
- Cannot share entities across pools
- Difficult to debug which pool is being used
- No practical benefit when thread isolation already works

**Future Scaling**:
- Need more capacity? Increase pool size (30 → 50 → 100)
- Need isolation? Already have it via thread pools
- True separate pools only make sense for: read replicas, different databases, or microservices

### Rate Limiting

**Purpose**: Prevent resource exhaustion and ensure fair usage

**Bulk API Rate Limits**:
- Per partner: Max 1 concurrent request
- Global: Max 10 concurrent requests
- Enforcement: Semaphore-based counting
- Response if exceeded: 429 Too Many Requests with Retry-After header

**Search API Rate Limits**:
- Per partner: 60 requests/minute
- Global: 200 requests/minute
- Enforcement: Token bucket (Guava RateLimiter)
- Response if exceeded: 429 Too Many Requests

**Event Ingestion Rate Limits**:
- Global: 100 events/minute
- Per application: 10 events/minute (prevent loops)
- Enforcement: Token bucket
- Response if exceeded: 429 (Core App should back off)

**Webhook Delivery Concurrency**:
- Per partner: Max 3 concurrent webhooks
- Global: Max 50 concurrent webhooks
- Enforcement: Semaphore per partner
- If exceeded: Queue for later delivery

### Query Optimization

**Database View Queries**:
- Ensure indexes exist on key columns
- Use batch fetching for bulk operations (IN clause with up to 1000 IDs)
- Consider pagination cursor strategy instead of offset (better performance at high page numbers)

**Elasticsearch Queries**:
- Index only searchable fields (~20 fields, not all 500)

## 10. Security

### Authentication & Authorization

**Partner Authentication**:
- Mechanism: Bearer Token (JWT or opaque token)
- Header: Authorization: Bearer {token}
- Partner identification: X-Partner-ID header (validated against token)
- Token storage: Encrypted in database
- Token rotation: Support token refresh without downtime

**Core App Authentication**:
- Mechanism: Shared secret or mutual TLS
- Purpose: Secure event ingestion endpoint
- Rotation: Coordinate with Core App team

**Admin API Authentication**:
- Mechanism: OAuth2 or internal SSO
- Purpose: Protect configuration and operational endpoints
- Roles: admin (full access), operator (read-only + limited actions)

### Partner Webhook Security

**Outbound Authentication**:
- Support multiple auth types: Bearer Token, Basic Auth, API Key, Custom Headers
- Store credentials encrypted (use Spring encryption or external vault)
- Per-partner configuration

**Request Signing** (Optional, Future):
- HMAC-SHA256 signature of payload
- Partner validates signature to ensure authenticity
- Include signature in custom header: X-Signature

**IP Whitelisting** (Optional, per partner):
- Partner provides allowed IP ranges for inbound API calls
- Enforce at API Gateway or application level
- Store in partner configuration


### Security Best Practices

**Input Validation**:
- Validate all API request payloads against schema
- Sanitize inputs to prevent injection attacks
- Limit request payload sizes (e.g., 1MB max)

**Dependency Management**:
- Keep all dependencies up to date
- Regular security scanning (Dependabot, Snyk)
- Apply security patches promptly

**Secrets Management**:
- Never commit secrets to version control
- Use environment variables or secrets manager
- Rotate secrets periodically

**Logging**:
- Never log sensitive data (passwords, tokens)
- Mask sensitive fields in logs
- Log all authentication failures for monitoring

### Logging Strategy

**Log Levels**:
- ERROR: System errors, unhandled exceptions, critical failures
- WARN: Recoverable errors, retry attempts, circuit breaker events
- INFO: Business events (event processed, webhook delivered), startup/shutdown
- DEBUG: Detailed flow, request/response bodies (non-production only)

**Structured Logging**:
- Use JSON format for log entries
- Include: timestamp, level, service, component, message
- Add context: partnerId, applicationId, eventId, correlationId
- Tool: Logback with JSON encoder

**Key Log Events**:
- Event received from Core App
- Event processing started/completed
- Webhook delivery attempted/succeeded/failed
- Circuit breaker state changes
- Configuration changes
- API requests (with response time)
- Errors with full context

**Log Aggregation**:
- Send logs to centralized system (ELK, Splunk, CloudWatch)
- Enable log search and analysis
- Set up log-based alerts

### Metrics Collection

**MVP Metrics** (Core 4 - MUST HAVE):

These metrics tell you if the system is healthy and functioning:

**1. Request Duration & Count** (Automatic from Spring Boot Actuator)
- Enabled by default with @Timed or Actuator
- Tracks: request count, latency (P50, P95, P99)
- Per endpoint automatically

**2. Event Processing Success/Failure**
- Counter: events.processed (labeled by status: success/failure)
- Tells you: Are events being processed? What's the success rate?

**3. Webhook Delivery Success/Failure per Partner**
- Counter: webhook.delivery (labeled by partner_id, status)
- Tells you: Which partners are receiving data successfully?

**4. Dead Letter Queue Size**
- Gauge: dead_letter.size (labeled by partner_id)
- Tells you: Are failures accumulating? Which partner has issues?

**Implementation Simplicity**:
- Use Micrometer (included in Spring Boot)
- Inject MeterRegistry into services
- Create counters/gauges where events happen
- Export via /actuator/prometheus
- Total implementation: ~50 lines of code

**Phase 2 Metrics** (POST-MVP, add if needed):
- Thread pool utilization per pool
- Connection pool utilization
- Circuit breaker state per partner
- Elasticsearch sync lag
- Rate limit exceeded count
- API request count by partner

**Why Prioritize**:
The core 4 metrics answer the critical questions:
- Is the system processing events? (metric 2)
- Are partners receiving data? (metric 3)
- Are there failures piling up? (metric 4)
- How fast are APIs responding? (metric 1)

Additional metrics add operational insight but aren't critical for MVP. Add them incrementally based on actual monitoring needs, not theoretical completeness.

## 15. Key Considerations & Risks

### Technical Risks

**Risk 1: Database View Performance**
- **Risk**: Views may be slow with complex joins and large data
- **Mitigation**: Work with Core App team to optimize view queries, add indexes, consider materialized views
- **Monitoring**: Track query execution times, alert if > 1 second

**Risk 2: Elasticsearch Sync Lag**
- **Risk**: 5-minute sync delay may not be acceptable for real-time search
- **Mitigation**: Start with 5-minute sync, evaluate if real-time needed, consider event-based sync
- **Monitoring**: Track sync lag metric, alert if > 10 minutes

**Risk 3: Partner Webhook Reliability**
- **Risk**: Partners may have unstable webhooks, causing high failure rates
- **Mitigation**: Comprehensive retry logic, circuit breakers, dead letter queue, partner SLA discussions
- **Monitoring**: Track per-partner success rates, alert on degradation

**Risk 4: Memory Pressure from Bulk Operations**
- **Risk**: Loading 1000+ records into memory for serialization may cause GC issues
- **Mitigation**: Response streaming (chunked transfer encoding), pagination with smaller page sizes, monitor heap usage
- **Monitoring**: Track heap utilization during bulk operations, GC pause times

**Risk 5: Core App Event Delivery Failure**
- **Risk**: If Integration Service is down, Core App dead letter grows without bounds
- **Mitigation**: Core App dead letter capacity planning, alerting, replay mechanisms, consider Kafka for future
- **Monitoring**: Track Core App dead letter size (coordinate monitoring with Core App team)
