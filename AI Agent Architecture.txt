## 1. Understanding Agents vs. Simple APIs

### Initial Scenario
**Proposed System:**
- Employee submits request
- Manager approval API (calls LLM, makes decision)
- HR approval API (calls LLM, makes decision)
- Workflow moves forward or rejects

### Key Question
**Can these APIs be called "agents"?**

**Answer: NO**

### What Makes a True Agent
An agent has:
- **Autonomy** - Ability to choose which actions to take
- **Goal-orientation** - Works toward an objective, not just responding to prompts
- **Tool use** - Can decide to use different tools/APIs as needed
- **Multi-step reasoning** - Can plan and execute multiple actions
- **Environmental interaction** - Observes results and adapts behavior

### Your APIs Are
**LLM-powered decision points** - Smart decision logic, but not autonomous agents

### How to Make it an Agent

**Option 1: Approval Agent (Single Agent)**
Instead of hardcoded workflow, one agent orchestrates:
- Determines required approvals based on request
- Calls appropriate approval checks
- Monitors process and escalates if needed

**Option 2: Investigative Agent**
- Gathers information before routing
- Searches policy database
- Checks employee history
- Identifies appropriate approval chain

**Option 3: Manager/HR Assistant Agents**
- Help managers make decisions
- Pull relevant data
- Draft recommendations with reasoning
- Present to human for final decision

## 2. When to Use RAG

### Use RAG When You Need

**1. Domain-Specific Knowledge**
- Company internal policies, procedures
- Product documentation
- Legal documents specific to your organization
- Medical records, patient histories
- Customer support knowledge bases

**2. Information Changes Frequently**
- Product catalogs with updates
- Price lists, inventory
- Regulatory changes
- Just update documents, no model retraining needed

**3. Source Attribution**
- Legal advice (cite specific statutes)
- Medical recommendations (cite studies)
- Compliance requirements
- Must show where information came from

**4. Large Knowledge Base**
- Information too large for context window
- Entire codebases
- Complete product manuals
- Years of company documents

**5. Reduce Hallucinations**
- Accuracy is critical
- Can't afford made-up information
- Grounds responses in actual documents

### Don't Use RAG When

**1. General Knowledge Tasks**
- LLM already knows the answer
- "What's the capital of France?"
- Basic programming syntax
- General science explanations

**2. Creative/Generative Tasks**
- Writing poetry, stories
- Brainstorming ideas
- Creating fictional content

**3. Reasoning/Math Problems**
- Logical deduction
- Calculations
- Algorithm solving

**4. Real-Time Data**
- Current stock prices → Use API
- Weather now → Use weather API
- Live sports scores → Use data API

**5. Personal/Contextual Conversation**
- Chat history
- User preferences
- Use conversation memory, not RAG

### Applied to Loan Approval

**Use RAG for:**
- Company lending policies
- Historical loan decisions
- Risk assessment guidelines
- Regulatory requirements
- Similar past cases

**Don't use RAG for:**
- Simple calculations (DTI ratio)
- Database queries (current budget)
- Real-time credit scores (use API)
- Workflow rules (use code logic)

## 3. What is RAG

### Definition
**RAG = Retrieval-Augmented Generation**

Combines:
1. **Retrieval** - Finding relevant information from knowledge base
2. **Generation** - Using LLM to generate answers based on that information

### Simple Analogy
**Open-book exam vs. Closed-book exam**

**Without RAG (Closed-book):**
- Relies only on what LLM memorized during training
- Might give wrong answers or say "I don't know"
- Can't access new information

**With RAG (Open-book):**
- Can look up information during answer generation
- Gives accurate answers based on documents
- Can reference latest information

### How RAG Works

**Simple Example:**
```
Question: "What's the approval limit for conference expenses?"

Without RAG:
→ LLM: "I don't have information about your company's 
   specific policies..."

With RAG:
Step 1: Search company documents
→ Finds: "Company Policy Section 4.2: Conference 
   attendance up to $3,500 requires manager approval only..."

Step 2: LLM generates answer using retrieved policy
→ "According to Company Policy Section 4.2, conference 
   expenses up to $3,500 require only manager approval..."
```

### Storage Mechanism

RAG converts documents into **vectors** (mathematical representations):
```
Document: "Loans under $10,000 with credit score above 680 
           are typically approved"
           ↓
Converted to vector: [0.23, 0.87, -0.12, 0.45, ...]
           ↓
Stored in Vector Database
```

### Why Vectors?
Enables **semantic search** - finding by meaning, not just keywords

**Traditional search:**
- Query: "remote work policy"
- Finds: Documents with exact words "remote work"
- Misses: "telecommuting," "work from home"

**RAG search:**
- Query: "remote work policy"
- Finds: "Remote work," "Telecommuting," "Work from home," "Flexible location"
- All have similar meaning, different words

### Complete RAG Flow

**Setup (One-time):**
1. Collect documents (policies, historical cases)
2. Convert to vectors (embeddings)
3. Store in Vector Database

**Query Time (Every question):**
1. User asks question
2. Convert question to vector
3. Search vector database for similar vectors
4. Retrieve top 3-5 most relevant documents
5. Send to LLM: question + relevant documents
6. LLM generates answer based on documents
7. Return answer to user

### Key Benefits

1. **Accuracy** - Answers from YOUR documents
2. **Up-to-date** - Update documents anytime
3. **Source Citation** - Can show where answer came from
4. **Reduced Hallucination** - Must use retrieved documents
5. **Domain-Specific** - Understands your terminology

## 4. Storing Historical Data in RAG

### Question
Can historical data be stored in RAG to help agent make decisions?

**Answer: Yes, absolutely!**

### Types of Historical Data

**1. Historical Cases (Perfect for RAG)**
- Past approval decisions
- Resolved tickets
- Completed workflows
- Unstructured or semi-structured data
- Need semantic search ("find similar cases")

**Example:**
```
Case #1234 (Approved)
Date: 2024-03-15
Employee: John Doe, Marketing
Request: $3,500 conference attendance (VidCon 2024)
Manager reasoning: "Aligns with Q2 marketing goals"
HR reasoning: "Within professional development budget"
Outcome: Approved
```

**2. Structured Metrics (Better in Database)**
- Aggregated stats
- Time-series data
- Numerical trends
- Need precise filtering/aggregation

**Example:**
```
Database Query:
SELECT AVG(amount) FROM approvals 
WHERE department='Marketing' 
AND request_type='Conference'
```

**3. Hybrid Approach (Best)**
Combine both:
- Database for numbers (totals, averages, counts)
- RAG for context (narratives, patterns, reasoning)

**Example Agent Decision:**
```
Step 1: Query database
- Employee spent $4,500 of $10,000 budget
- Department average: $3,200 per request

Step 2: Search RAG
- Similar training requests approved
- Pattern: Training >$5K requires business case

Step 3: Agent decides
"Request above average but within budget. Historical 
cases show approval when ROI demonstrated..."
```

### When to Use RAG vs Database

| Data Type | RAG | Database |
|-----------|-----|----------|
| Case narratives | ✅ Perfect | ❌ Poor |
| Approval reasoning | ✅ Perfect | ❌ Poor |
| Similar decisions | ✅ Perfect | ⚠️ Limited |
| Policy interpretations | ✅ Perfect | ❌ Poor |
| Numeric totals | ❌ Poor | ✅ Perfect |
| Counts/aggregations | ❌ Poor | ✅ Perfect |
| Time-series trends | ❌ Poor | ✅ Perfect |
| "Find similar to X" | ✅ Perfect | ❌ Poor |

## 5. RAG vs Traditional Databases

### Key Difference

**RAG is not a database itself** - it's a technique that uses specialized databases

### Traditional Database
```
Stores: Structured data in tables/documents
Search: Exact matching, SQL queries
Format: Rows/columns or JSON

Example:
SELECT * FROM employees WHERE salary > 50000
→ Returns exact matches
```

### Vector Database (Used by RAG)
```
Stores: Data as mathematical vectors (embeddings)
Search: Semantic similarity
Format: High-dimensional vectors + original text

Example:
Query: "expensive training courses"
→ Returns semantically similar:
  - "high-cost professional development"
  - "premium certification programs"
  - "$5000+ skill enhancement"
```

### Yes, RAG Data IS Stored in a Database

But it's a **different type** - Vector Database

**Vector Databases:**
- Pinecone
- Weaviate
- Qdrant
- Chroma
- FAISS
- Milvus

### Complete Architecture

```
PostgreSQL (Traditional DB)
├─ document_id: 12345
├─ text: "Conference policy states..."
├─ created_date: 2024-01-15
└─ category: "expense_policy"

Pinecone (Vector DB)
├─ vector_id: 12345 (links to Postgres)
├─ embedding: [0.23, 0.87, -0.12, ...]
└─ metadata: {category: "expense_policy"}

When User Queries:
1. Convert query to vector
2. Search Pinecone for similar vectors
3. Get document IDs
4. Fetch full text from PostgreSQL
5. Send to LLM for answer generation
```

### Why Can't You Use Traditional DB for RAG?

**Problem: Traditional DBs don't understand meaning**

```
Query: "What's the policy on work from home?"

Traditional DB:
SELECT * FROM policies WHERE text LIKE '%work from home%'
Results:
✓ "Work from home policy..."
✗ Misses: "Remote work guidelines"
✗ Misses: "Telecommuting procedures"

Vector DB:
Query embedding: [0.12, 0.56, -0.34, ...]
Results (ranked by similarity):
1. "Remote work guidelines" (0.95)
2. "Work from home policy" (0.93)
3. "Telecommuting procedures" (0.89)
4. "Flexible location arrangements" (0.87)
```

### Practical: Use BOTH

```
PostgreSQL (Traditional DB)
- Employee records (structured)
- Approval history (structured)
- Budget tracking (numbers)
- Workflow state

Pinecone (Vector DB for RAG)
- Company policies (unstructured)
- Historical case narratives
- Approval reasoning examples
- Exception patterns
```

## 6. How Agents Access Databases

### Multiple Options

**Option 1: Direct Database Connection**
Agent connects directly using database drivers

**Pros:**
- Simple, direct access
- Fast performance
- No API overhead

**Cons:**
- Security risk (agent has DB credentials)
- Hard to monitor/audit
- Tight coupling
- Difficult to scale

**When to use:** Small, internal systems

---

**Option 2: Tool/Function Calling**
Agent has tools it can call, LLM decides when

**Pros:**
- Agent autonomously decides what data needed
- More flexible and intelligent
- Can combine multiple data sources

**Cons:**
- More complex to implement
- Still requires DB access in code

**When to use:** Intelligent data retrieval needed

---

**Option 3: API Layer (Recommended)**
API sits between agent and databases

**Architecture:**
```
Agent
  ↓ HTTP/REST
API Layer ← Authentication, Rate limiting, Logging
  ↓
  ├─→ PostgreSQL
  └─→ Vector DB
```

**Pros:**
- Security (API handles auth)
- Monitoring (track all queries)
- Rate limiting
- Decoupling
- Versioning
- Caching

**Cons:**
- Extra latency
- More infrastructure
- Additional development

**When to use:** Production systems, multiple agents

---

**Option 4: SQL Agent**
LLM generates SQL queries from natural language

**Pros:**
- Very flexible
- No pre-defined queries
- Natural language interface

**Cons:**
- Security risk (can execute any SQL)
- Hallucination risk
- Unpredictable performance
- Need read-only access

**When to use:** Internal analytics, trusted environments

---

**Option 5: Hybrid (Best for Loan System)**

```
Agent:
├─ Direct access: Vector DB (fast semantic search)
└─ API calls: PostgreSQL (secure, auditable)
```

For your loan approval:
- API for sensitive data (employee records, budgets)
- Direct for RAG (fast, low-risk read-only)

## 7. Spring Events for Integration

### Spring Events Overview

Allows internal communication within Spring applications

**Two Types:**

### 1. Application Events (Single Application)

For when Agent and Loan App in SAME Spring Boot application

**Event Class:**
```java
public class LoanSubmittedEvent extends ApplicationEvent {
    private String loanId;
    // constructor, getters
}
```

**Publisher:**
```java
@PostMapping("/api/loan/submit")
public ResponseEntity<?> submitLoan(@RequestBody LoanApplication loan) {
    loanRepository.save(loan);
    eventPublisher.publishEvent(new LoanSubmittedEvent(this, loan.getId()));
    return ResponseEntity.ok("Loan submitted, analysis in progress");
}
```

**Listener:**
```java
@EventListener
@Async
public void handleLoanSubmission(LoanSubmittedEvent event) {
    String loanId = event.getLoanId();
    LoanApplication loan = loanRepository.findById(loanId);
    AgentRecommendation recommendation = analyzeLoan(loan);
    recommendationRepository.save(recommendation);
}
```

---

### 2. Spring Cloud Stream (Distributed Applications)

For when Agent and Loan App are SEPARATE microservices

**Loan App publishes to Kafka/RabbitMQ**
**Agent consumes from message broker**

**application.yml:**
```yaml
spring:
  cloud:
    stream:
      bindings:
        loan-submitted:
          destination: loan-events
          binder: kafka
```

---

### Which to Use?

| Scenario | Recommendation | Why |
|----------|---------------|-----|
| Agent part of Loan App | Application Events | Simple, no external infrastructure |
| Agent separate microservice | Spring Cloud Stream + Kafka | Proper decoupling, scalability |
| Starting simple | Application Events | Easy to migrate later |
| Need guaranteed delivery | Spring Cloud Stream | Message broker ensures delivery |
| High volume (1000s/day) | Spring Cloud Stream | Better scalability |

---

### Your Flow with Spring Cloud Stream

```
Step 1: Loan App Publishes Event
POST /api/loan/submit
→ Save to DB
→ Publish to Kafka: loan-events topic

Step 2: Agent Consumes Event
→ Receives loan_id
→ Calls Loan App API: GET /api/loan/{loan_id}
→ Analyzes loan
→ Calls Loan App API: POST /api/loan/{loan_id}/recommendation

Step 3: Loan App Receives Recommendation
→ Saves to database
→ Updates loan status: ANALYSIS_COMPLETE

Step 4: Loan Officer Opens Application
→ Sees pre-populated recommendation
→ Can chat with agent
```

**Recommendation: Spring Cloud Stream** for your described architecture (separate agent service)

## 8. RAG Memory Updates

### The Challenge

```
Initial: RAG has 1,000 historical loans (2023-2024)

New decisions:
Jan 27: Agent approves Loan #12345
Jan 28: Agent rejects Loan #12346
...

Question: How do new decisions get into RAG?
```

### Strategy 1: Immediate Update (Real-time)

**Update RAG immediately after decision finalized**

**Event-Driven:**
```
1. Loan officer finalizes decision
2. Save to database
3. Publish event: DecisionFinalizedEvent
4. RAG Updater Service listens
5. Formats decision as document
6. Generates embedding
7. Inserts into vector database
```

**Pros:**
- Immediate availability
- Always up-to-date
- Simple to implement

**Cons:**
- Small overhead per decision

---

### Strategy 2: Batch Update (Scheduled)

**Update RAG in batches at intervals**

```
Schedule: Every hour / nightly / weekly

Process:
1. Query DB for new decisions since last sync
2. Format all as documents
3. Generate embeddings in batch
4. Bulk insert to vector DB
```

**Pros:**
- Efficient batch processing
- Less overhead on main app

**Cons:**
- Delay before available

---

### Strategy 3: Dual-Write Pattern

**Write to both database AND vector DB simultaneously**

**Pros:**
- Strong consistency
- Immediate availability

**Cons:**
- Tighter coupling
- If vector DB slow, affects main flow

---

### Strategy 4: Change Data Capture (CDC)

**Automatically capture DB changes**

```
PostgreSQL → Debezium CDC → Kafka → RAG Sync Consumer
```

**Pros:**
- Zero code changes in main app
- Guaranteed to capture all changes
- Scalable

**Cons:**
- More complex infrastructure
- Overkill for simple cases

---

### Recommended: Strategy 1 (Event-Driven)

**Why:**
- Fits your Spring Events architecture
- Immediate consistency
- Clear audit trail
- Simple implementation

**With optimization:**
- Async processing (don't block finalization)
- Retry logic (if vector DB fails)
- Idempotency (prevent duplicates)
- Audit tracking

---

### Handling Updates vs New Inserts

**What if decision changes after added to RAG?**

**Option A: Update-in-Place (Recommended)**
```
When decision changes:
- Generate new document with updated info
- Upsert (replace existing entry with same ID)
```

**Option B: Keep History**
```
Keep versions: LOAN-12345_v1, LOAN-12345_v2
Metadata: version, is_latest
Search filter: only latest versions
```

---

### RAG Lifecycle Management

**Problem: RAG grows infinitely**

**Solution: Archival Strategy**
```
Scheduled job:
- Find decisions older than 3 years
- Move to archive collection
- Or delete if no longer relevant
- Keep metadata for audit
```

## 9. System Architecture Design

### Your Final Architecture

**Complete Flow:**

```
Step 1: User Submits Application
↓
Step 2: App Sends Event to Agent (loan_id)
↓
Step 3: Agent Calls GET API
        GET /api/loan/{loan_id}
        Returns: Complete application data
↓
Step 4: Agent Analyzes
        - Checks credit, income, debt ratio
        - Searches RAG (policies, similar cases)
        - Calculates risk score
        - Generates decision + rationale
↓
Step 5: Agent Calls POST API
        POST /api/loan/{loan_id}/recommendation
        Body: {decision, rationale, risk_score, ...}
↓
Step 6: Loan App Persists Decision
        - Saves to database
        - Updates application status
↓
Step 7: Loan Officer Opens Application
        - Sees pre-populated decision
        - Sees rationale
        - Can chat with agent
        - Can accept/modify/reject
```

### Integration Pattern

**Event-Driven Architecture (Recommended)**

```
Loan App → Spring Cloud Stream → Kafka → Agent Service
                                          ↓
                                     GET /api/loan
                                          ↓
                                      Analyze
                                          ↓
                                     POST /api/recommendation
                                          ↓
Loan App ← Receives & Persists ← ← ← ←
```

**Benefits:**
- Clean separation of concerns
- Asynchronous processing
- Scalable
- Flexible

## 11. Sensitive Data Handling

### Critical Requirement

**Agent must:**
- ✅ SEE sensitive data for analysis (SSN, accounts, addresses)
- ❌ NOT mention it in user-facing text

### Examples

**Agent CAN access:**
- Full SSN: 123-45-6789
- Exact salary: $73,247.89
- Bank account numbers
- Detailed credit reports

**Agent MUST NOT write:**
```
❌ "Borrower's SSN 123-45-6789 shows clean history"
❌ "Account #9876543210 has balance of $15,234"
```

**Agent SHOULD write:**
```
✅ "Borrower's credit history shows no delinquencies"
✅ "Verified sufficient liquid assets"
```

### Implementation Approaches

**Option 1: System Prompt Enforcement (Preferred)**

```
System prompt:
"You have access to sensitive data for analysis. 
NEVER include in responses:
- SSNs, account numbers
- Exact addresses, phone numbers
- Medical details
Use generalized terms:
- 'Credit history verified'
- 'Income bracket: $70K-$80K'
- 'Employment verified'"
```

**Option 2: Post-Processing Filter**
- Agent generates response
- System scans for sensitive patterns
- Redacts before showing user
- Logs violations

**Option 3: Dual-Agent Pattern**
- Analysis Agent: Full access, internal report
- Communication Agent: Sanitized user-facing text

### Validation Layer

Before showing response:
- Scan for SSN patterns (###-##-####)
- Scan for account numbers (10+ digits)
- Scan for addresses
- Flag if found → Block or auto-redact
- Log violation for review

## 12. Version Management & Resubmission

### The Challenge

```
Day 1, 10:00 AM: Borrower submits
  - Income: $75,000
  - Credit: 680

Day 1, 10:15 AM: Agent analyzes
  - Recommendation: APPROVE

Day 1, 3:00 PM: Borrower resubmits with changes
  - Income: $70,000 (corrected)
  - Credit: 685 (fresh pull)

Challenge: Agent must analyze V2 only, ignore V1
```

### Critical Design Decisions

**1. Version Tracking**
```
Application Versions:
├─ Version 1 (original)
│  ├─ Data snapshot
│  ├─ Agent recommendation
│  └─ Status: SUPERSEDED
│
└─ Version 2 (resubmission)
   ├─ Data snapshot
   ├─ Agent recommendation
   └─ Status: CURRENT
```

**2. Agent Analysis Invalidation**

When borrower resubmits:
1. Invalidate old analysis
2. Trigger new analysis event
3. Clear old recommendation
4. Preserve history for audit

**3. Conversation Handling**

**Options:**
- **Auto-terminate old conversation** (mark read-only)
- **Start new conversation** for new version
- **Link versions** for context

**Recommended: Hybrid**
- Preserve V1 conversation (read-only)
- Start V2 conversation automatically
- Link: "Previous version discussion available"

**4. Data Snapshot Strategy**

**Critical Rule: Agent ONLY sees version it's analyzing**

```
When analyzing:
├─ Create immutable snapshot
├─ Tag with version ID
├─ Agent references this snapshot only
└─ New submission = new snapshot = new analysis
```

**5. Database Schema**

```
applications:
├─ id: LOAN-12345
├─ current_version: 2
└─ status: PENDING_REVIEW

application_versions:
├─ application_id: LOAN-12345, version: 1
│  ├─ data: {income: 75000, credit: 680}
│  └─ superseded: true
│
└─ application_id: LOAN-12345, version: 2
   ├─ data: {income: 70000, credit: 685}
   └─ superseded: false (current)

agent_recommendations:
├─ application_id: LOAN-12345, version: 1
│  ├─ decision: APPROVE
│  └─ valid: false (invalidated)
│
└─ application_id: LOAN-12345, version: 2
   ├─ decision: REJECT (new)
   └─ valid: true (current)
```

**6. Event Flow**

```
Resubmission Event:
├─ Increment version
├─ Save V2 snapshot
├─ Mark V1 superseded
├─ Invalidate V1 recommendation
├─ Archive V1 conversation
├─ Notify officer
└─ Trigger agent re-analysis
```

**7. Critical Rule**

**Agent NEVER mixes versions**

```
❌ Wrong:
"In V1 income was $75K, now $70K, consider both"

✅ Correct:
"Analyzing V2. Income: $70K. No knowledge of previous versions."

Enforcement:
- Agent receives ONLY current version data
- No access to version history
- System prompt: "Analyze provided data only"
```

**8. RAG Update Strategy**

**Recommended: Store all versions**
```
RAG entries:
├─ LOAN-12345-V1 (status: SUPERSEDED)
└─ LOAN-12345-V2 (status: FINAL)

Search filter: status = "FINAL"
```

Benefits:
- Complete history
- Learn from corrections
- Audit trail

## 13. Questions for Agent Team

### Critical Questions from App Team Perspective

**1. API Contract & Integration**

## 15. Chat Implementation & Caching

### Architecture Clarification

**Chat window in Loan App:**
- UI owned by App Team
- Displayed when LO opens loan
- UI calls App Service (not Agent directly)

### Flow

```
UI (Browser)
    ↓ "What's borrower's income?"
    ↓
App Service (Your backend)
    ↓ Fetch conversation history from YOUR DB
    ↓ Add loan context
    ↓ Call Agent API
    ↓
Agent Service
    ↓ Returns answer
    ↓
App Service
    ↓ Save to YOUR DB
    ↓ Return to UI
    ↓
UI displays response
```

### Who Caches Conversation?

**App Team caches, not Agent Team**

**Why:**
- App controls the session
- App knows context (loan, version, officer)
- App can optimize (browser, server-side, or both)
- Agent stays stateless

### Caching Strategy

**Browser (temporary, current session):**
- Conversation while tab open
- sessionStorage or component state
- Cleared when navigate away

**Your Database (permanent):**
- Every message ever sent
- Every agent response
- Timestamps, who asked what
- Persists forever (or per retention policy)

**Agent's side (optional):**
- Might keep copy for audit
- But you don't rely on them
- You fetch from YOUR database

### Opening Old Chat After 10 Days

**What happens:**
1. LO clicks loan
2. Your app: `SELECT * FROM chat_messages WHERE loan_id = '12345'`
3. Load into UI
4. LO sees complete history
5. LO types new message
6. Your backend must send to agent:
   - Current question
   - Full conversation history
   - Loan context

### Agent Has No Memory

**Agent is stateless between calls**

Every API call must include:
```
POST /agent/chat
{
  "loan_id": "LOAN-12345",
  "conversation_history": [
    {"role": "user", "message": "Why approve?", "timestamp": "10 days ago"},
    {"role": "agent", "message": "Based on credit 720...", "timestamp": "10 days ago"},
    ...
  ],
  "current_question": "What if income $5K higher?"
}
```

### Conversation Management Options

**Option 1: Send Full History (Recommended)**
- Your app sends all previous messages
- Agent has complete context
- Simple for agent (stateless)
- Cons: Large payload if 100+ messages

**Option 2: Agent Stores (More Complex)**
- Send conversation_id only
- Agent looks up history on their side
- Smaller payload
- Cons: Agent must store, syncing issues

**Hybrid (Most Common):**
- Your DB = source of truth
- Send recent messages (last 20)
- Summarize older messages if needed

## 16. Frontend Technology Choice

### Angular for Chat?

**Will it work?** Yes

**Is it optimal?** No

### Challenges with Angular

1. **Real-time updates** - Heavy change detection for frequent messages
2. **Performance** - Two-way binding overhead with 100+ messages
3. **Development speed** - More verbose than React
4. **Libraries** - Fewer mature chat libraries

### Better Alternative: React

**Why React for Chat:**
- Lighter weight
- Rich ecosystem (react-chat-elements, wavesurfer.js)
- Better real-time performance
- Faster development

### Micro-Frontend Approach

**Build React chat as separate component, embed in Angular app**

**Implementation: Web Components (Cleanest)**

```
Angular sees it as:
<chat-widget 
  loan-id="12345"
  (onClose)="closeChat()"
  (onMessage)="handleMessage($event)">
</chat-widget>
```

React chat compiled as custom HTML element.

### Communication

**Angular → React:**
Pass data via props/attributes

**React → Angular:**
Emit events

**Both → Backend:**
Call same App Service APIs

### Complexity

**Medium** - Not trivial, but doable

**Worth it when:**
- Chat is complex (file upload, voice, video expected)
- Performance matters
- React team available
- Long-term investment

### Your Requirements

**Q&A + File Uploads + Voice Memos**

**Verdict: Absolutely go with React micro-frontend**

**Why:**
- File upload libraries (react-dropzone)
- Voice recording (react-mic, wavesurfer.js)
- Better performance for media
- React ecosystem saves months

**Time estimate:**
- Angular approach: 4-6 months, mediocre UX
- React micro-frontend: 2 months, excellent UX
- Extra cost: Learning micro-frontend (one-time)

**ROI: Absolutely worth it**

## 17. Multi-Role Agent Design

### Scenario

- 1 Loan Officer
- 20+ Risk Officers (credit, compliance, legal, fraud, etc.)
- Thousands of Borrowers

### Question: One Agent or Multiple?

**Answer: One agent with role-based context**

### Why One Agent

**Same capabilities needed:**
- Access loan data
- Search policies
- Analyze historical cases
- Chat with users

**Different contexts per role:**
- Loan Officer: Lending policies, approval history
- Credit Risk: Risk models, default patterns
- Compliance: Regulations, violations history
- Borrower: Public info, educational content

### Implementation

**One agent, role passed in API:**

```
POST /agent/analyze
{
  "loan_id": "12345",
  "role": "CREDIT_RISK_OFFICER",
  "user_id": "risk_officer_123"
}
```

Agent adjusts:
- System prompt (personality, tone)
- RAG filtering (relevant documents only)
- Response style

### Agent Personalities

**Different branding, same backend:**

```
Loan Officer sees:
💼 LendingPro Assistant
"Hi! I'm here to help with approvals..."

Credit Risk Officer sees:
🛡️ RiskGuard Analyst
"Let me assess the risk profile..."

Borrower sees:
💬 LoanHelper
"I'm here to help with your application..."
```

### Context Management Challenge

**Can one agent handle 20+ different contexts effectively?**

**Answer: Yes, with context partitioning**

### Solution: RAG Collections

**Separate knowledge domains:**

```
RAG Structure:
├─ lending_policies (for loan officers)
├─ credit_risk (for credit risk officers)
├─ compliance (for compliance officers)
├─ legal_risk (for legal officers)
├─ fraud_detection (for fraud analysts)
├─ borrower_faq (for borrowers)
```

**When processing request:**
```
Credit Risk Officer asks: "What's the default risk?"

Agent logic:
1. Role: CREDIT_RISK_OFFICER
2. Search ONLY: credit_risk collection
3. Ignore: compliance, legal, borrower_faq
4. Generate technical risk response
```

### Role-Based Access Control

**Critical for borrowers:**

**Officers (full access):**
- All loan data
- Internal policies
- Risk scores
- Historical cases

**Borrowers (restricted):**
- Only THEIR loan
- Public-facing info
- Application status
- No internal scores/notes

### System Prompt Per Role

**Credit Risk Officer:**
```
"You are RiskGuard, a technical risk analyst.
Focus on PD, LGD, risk scores.
Use technical terminology.
DO NOT discuss compliance or lending strategy."
```

**Borrower:**
```
"You are LoanHelper, a friendly assistant.
Use simple language.
Provide educational content.
DO NOT reveal internal risk scores or officer notes.
DO NOT discuss other applicants."
```

### Preventing Context Confusion

**Context partitioning prevents:**
- Mixing credit risk with operational risk
- Giving borrowers internal information
- Legal advice when asking about lending

**Enforcement:**
- Role-to-collection mapping
- Filtered RAG searches
- Focused system prompts

### Example: Same Question, Different Roles

**"Is this risky?"**

**Loan Officer:**
```
LendingPro: "This loan looks solid! Credit 720 
is great, DTI 32% is comfortable. Low risk. 👍"
```

**Credit Risk Officer:**
```
RiskGuard: "Risk Assessment:
- PD: 2.3%
- LGD: 45%
- Expected Loss: $1,150
- Grade: B+ (Low-Medium)"
```

**Borrower:**
```
LoanHelper: "Your application meets our lending 
criteria. We'll review and respond within 3 business 
days. Need help with anything else?"
```

### Scalability

**One agent handles all users because:**
- Stateless (each request independent)
- Role-based filtering (access control)
- RAG partitioning (relevant context only)
- LLM scales horizontally

### Cost Optimization

**More users = more cost:**

**Strategies:**
- Officers: Unlimited (business critical)
- Borrowers: Rate limited (5 messages/day)
- Officers: GPT-4 (complex)
- Borrowers: GPT-3.5 (simple FAQs)
- Cache common borrower questions

### When Multiple Agents Needed

**Only if:**
- Different LLM models entirely
- Separate infrastructure/security
- Different vendors
- Completely different workflows

**For your case: One agent, multiple contexts = sufficient**

## Summary of Key Decisions

### Architecture
- ✅ **Event-driven**: Spring Cloud Stream + Kafka
- ✅ **API Layer**: Between agent and databases
- ✅ **One Agent**: Role-based context, not multiple agents

### Data Management
- ✅ **Hybrid storage**: Traditional DB + Vector DB (RAG)
- ✅ **Immediate RAG updates**: Event-driven after finalization
- ✅ **Version snapshots**: Immutable, agent sees only assigned version

### Integration
- ✅ **Loan App triggers**: Send event with loan_id
- ✅ **Agent fetches data**: Calls GET /api/loan/{id}
- ✅ **Agent posts back**: POST /api/loan/{id}/recommendation
- ✅ **App persists**: Saves to database, ready for officer

### Sensitive Data
- ✅ **Agent sees all**: Needs full data for analysis
- ✅ **Agent doesn't mention**: System prompt + output validation
- ✅ **Generalized language**: "Income verified" not "$73,247.89"

### Chat Implementation
- ✅ **App Service mediates**: UI → App → Agent, not direct
- ✅ **App caches**: Conversation stored in app database
- ✅ **Agent stateless**: Receives history with each request
- ✅ **React component**: Micro-frontend for rich chat (file upload, voice)

### Multi-Role Support
- ✅ **One agent**: Multiple personalities
- ✅ **Context partitioning**: Separate RAG collections
- ✅ **Role-based access**: Borrowers see limited data
- ✅ **Focused prompts**: Keep agent in appropriate context
